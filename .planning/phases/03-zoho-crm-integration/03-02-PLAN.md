# Plan 03-02: Record Mappers & Sync Service

Implement the mapping logic and the core synchronization service to push data from local storage to Zoho CRM.

## Problem Statement

We need a structured way to transform SQLModel database objects into the specific JSON payloads expected by the Zoho CRM `upsert` API. This logic must handle field mapping, data type conversions (e.g., Decimal to Float), and relationship mapping (linking Bids to Projects and Companies using their Zoho IDs).

## Proposed Changes

### 1. Entity Mappers (`src/integrations/zoho/mapper.py`)
- `to_zoho_project(project: Project)`
- `to_zoho_company(company: Company)`
- `to_zoho_bid(bid: Bid, project_zoho_id: str, company_zoho_id: str)`
- `to_zoho_error(error: SystemError)`

### 2. Sync Service (`src/integrations/zoho/sync_service.py`)
- `ZohoSyncService` class containing:
    - `sync_companies(session)`: Finds unsynced or modified companies, batches them, and upserts them to Zoho. Updates `zoho_id` in local DB.
    - `sync_projects(session)`: Identifies new projects and upserts them.
    - `sync_bids(session)`: Identifies new bids, requires linked project and company to have Zoho IDs.
    - `sync_all(session)`: Orchestrates the sync order (Companies -> Projects -> Bids).

## Task Breakdown

- [ ] **Task 1: Implement Record Mappers**
    - Map database fields to Zoho API names (per PRD).
    - Handle relationship linking for Bids (lookup format: `{"id": "..."}`).
- [ ] **Task 2: Implement Sync Service Logic**
    - Batching logic (Zoho supports up to 100 records per request).
    - Idempotency via `upsert`.
    - Handle and log sync individual record errors within a batch.
- [ ] **Task 3: Test Mapping and Batching**
    - Create `tests/test_zoho_sync.py` with mocked API responses.
    - Verify mapping and error handling on partial batch failures.
